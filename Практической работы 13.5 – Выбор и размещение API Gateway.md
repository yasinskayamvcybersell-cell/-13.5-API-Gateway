# Решение практической работы 13.5 – Выбор и размещение API Gateway

## Контекст
На основе предыдущих работ (2.6–12.5) спроектирована распределённая система микросервисов (Orders, Inventory, Pricing, Payments, Shipping, Cart, Notifications, Audit, Catalog, ExperimentProcessor). Внедрены:
- **API Gateway** для внешних пользователей (JWT, rate limiting, WAF, mTLS) – из задания 12.5.
- Событийная шина (Kafka), саги, OpenAPI.

**Новые требования**:  
Появились IoT устройства (станки, датчики контроля качества, системы учёта движения на складах), а также растёт сеть дистрибьюторов и пользователей. Необходимо собирать и обрабатывать телеметрию, принимать решения на основе данных, избегая хаоса в управлении потоками.

#### Обоснование размещения шлюзов
| Шлюз | Обоснование |
|------|-------------|
| **Public API Gateway** | Уже существующий (из 12.5). Обслуживает B2C клиентов (веб, мобильные приложения). Расположен в DMZ, так как доступен из интернета. |
| **B2B API Gateway** | Выделен отдельно, потому что дистрибьюторы имеют иные шаблоны трафика (пакетные заказы, более высокие лимиты, требуется API-ключ вместо JWT). |
| **IoT API Gateway** | Новый компонент для приёма телеметрии. Расположен в DMZ, поддерживает MQTT/WebSocket. Изолирует высоконагруженный поток от остальных сервисов. |

#### Почему не один шлюз?

- Разные протоколы (HTTP/REST vs MQTT/WebSocket).
- Разные SLA (для пользователей задержка <200 мс, для телеметрии – пропускная способность).
- Разные механизмы безопасности (JWT+2FA vs API Key vs X.509).
- Риск DDoS на публичном шлюзе не должен влиять на приём телеметрии.

## 2. Функционал и качественные характеристики API Gateway
### 2.1. Public API Gateway (B2C)
**Тип:** Обратный прокси + Edge Gateway (синхронный REST)

| Функция | Реализация | Качественные характеристики |
|---------|------------|-----------------------------|
| **Аутентификация** | JWT (Access + Refresh), валидация через Auth Service | <10 мс (95-й перцентиль) |
| **Rate limiting** | По IP и userId: 100 req/min для пользователя, 1000 req/min для IP | Точность ±5% |
| **Маршрутизация** | Правила paths: /api/v1/orders/* → Orders Service | <2 мс |
| **Кэширование** | Кэш GET-ответов (каталог, цены) Redis, TTL 30 с | Hit ratio >60% |
| **WAF** |	ModSecurity + OWASP CRS | Обнаружение 99% типовых атак |
| **CORS** | Белый список доменов | <1 мс |
| **Логирование** |	Все запросы (без ПДн) в ELK | Асинхронно |
| **Мониторинг** | Prometheus (RPS, latency, ошибки) | Экспорт каждые 15 с |

**Цифровые требования:**
- **Пропускная способность:** до 10 000 RPS (пик)
- **Latency p99:** <100 мс (внутри шлюза)
- **Доступность:** 99.99% (кластер 4 ноды)
- **Масштабирование:** горизонтальное по CPU

### 2.2. B2B API Gateway (дистрибьюторы)
**Тип:** Шлюз с усиленной безопасностью и поддержкой пакетных операций

| Функция | Реализация | Качественные характеристики|
|---------|------------|----------------------------|
| **Аутентификация** | API Key + mTLS. Ключи в Vault | <20 мс |
| **Rate limiting** | По API Key: 5000 req/min |Точность ±2% |
| **Валидация bulk-запросов** |	Приём JSON-массивов (до 1000 заказов) |	<500 мс на 1000 элементов |
| **Авторизация** |	OPA (проверка доступа к складам) | <30 мс |
| **Трансформация**	| XML → JSON (для старых систем дистрибьюторов)	| <200 мс |
| **Квотирование**	| Ежемесячные лимиты в Redis | Сброс 1-го числа |

**Цифровые требования:**
- **Пропускная способность:** до 2000 RPS
- **Latency p99:** <200 мс (с конвертацией)
- **Доступность:** 99.95%
- **Максимальный размер запроса:** 10 МБ

### 2.3. IoT API Gateway
**Тип:** Протокольный шлюз (MQTT/WebSocket) + асинхронный буфер

| Функция | Реализация | Качественные характеристики |
|---------|------------|-----------------------------|
| **Приём телеметрии** | MQTT (QoS 1), WebSocket | До 50 000 сообщений/с |
| **Аутентификация устройств** | X.509 сертификаты + CRL | <50 мс (с кэшем) |
| **Валидация и дедупликация** | Проверка JSON-схемы, дедупликация deviceId+timestamp за 10 с | 100% дубликатов отбрасываются |
| **Буферизация** | Буфер → Kafka, при переполнении – диск | Буфер на 1 млн сообщений (~5 ГБ) |
| **Преобразование** | Унификация телеметрии в канонический Avro | <10 мс на сообщение |
| **Маршрутизация команд** | Из Kafka → устройству через MQTT | Задержка <500 мс |
| **Health check** | Keepalive, алерт при пропаже >5 минут | Проверка каждые 10 с | 

**Цифровые требования:**
- **Пропускная способность:** 50 000 сообщений/с (пик), 10 000 в среднем
- **Размер сообщения:** до 4 КБ (стандарт), до 1 МБ (редко)
- **Latency p99 (устройство → Kafka):** <100 мс
- **Доступность:** 99.95% (кластер 3 ноды)
- **Гарантии доставки:** At-least-once, идемпотентность на бэкенде

## 3. Избегаемые антипаттерны
| Антипаттерн | Почему не допущен |
|-------------|-------------------|
| **"Gateway монолит"** (один шлюз на всё) | Разные протоколы, SLA, безопасность – выделены отдельные шлюзы. |
| **"Умный шлюз"** (бизнес-логика в шлюзе)	| Только маршрутизация, аутентификация, лимиты; бизнес-логика в микросервисах. |
| **"Внутренний шлюз для внешних вызовов"** | Все внешние вызовы через DMZ-шлюзы, внутренние сервисы общаются напрямую (mTLS/Kafka). |
| **"Толстый клиент"** | Клиенты ходят только через шлюз, не вызывают микросервисы напрямую.| 

## 4. Итоговое описание связей (текстовое)
- **Пользователи (B2C)** → Public API Gateway (HTTPS) → микросервисы (Orders, Catalog, Pricing, Cart)
- **Дистрибьюторы (B2B)** → B2B API Gateway (HTTPS + mTLS) → микросервисы (Orders, Inventory, Shipping) с повышенными лимитами
- **IoT устройства** → IoT API Gateway (MQTT/WebSocket) → буферизация → Kafka → Telemetry Processor
- **Telemetry Processor** → при необходимости команды через IoT Gateway к устройствам
- Все шлюзы используют **Vault** (секреты) и **OPA** (авторизация)
- Внутренний трафик микросервисов – mTLS и/или Kafka, **без прохода через шлюзы** (во избежание лишней задержки)

## Заключение
Предложенная архитектура с тремя специализированными API Gateway обеспечивает:
- Изоляцию потоков разного характера (B2C, B2B, IoT)
- Соблюдение требований безопасности (JWT, API Keys, X.509)
- Масштабируемость под растущую нагрузку (пиковые 50 000 сообщений/с от IoT)
- Централизованное управление политиками (rate limiting, квоты, авторизация)

Каждый шлюз имеет минимально необходимый набор функций, измеримые качественные характеристики и избегает антипаттернов.
